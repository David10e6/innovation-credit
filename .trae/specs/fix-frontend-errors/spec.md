# 前端错误修复 - 产品需求文档

## 概述
- **摘要**：修复学生端、辅导员端和管理员端的前端错误，包括数据渲染、权限控制、API调用和表单验证等问题。
- **目的**：确保前端与后端正确对接，实现角色权限控制，保证数据安全和功能正常。
- **目标用户**：学生、辅导员、管理员。

## 目标
- 学生端：删除本地假数据，调用真实API，只显示本院和校级通知，移除操作按钮。
- 辅导员端：携带院系ID请求数据，只显示本院通知，修复数据绑定和分页逻辑。
- 管理员端：携带校级标识发布通知，只显示校级通知，修复数据绑定和分页逻辑。
- 统一三端API路径、请求方式、参数字段与后端一致，完善表单校验，修复分页参数。

## 非目标（范围外）
- 不修改后端API实现。
- 不添加新功能，只修复现有错误。
- 不修改UI设计，只修复功能逻辑。

## 背景与上下文
- 前端使用Vue 3 + Element Plus开发。
- 后端API已存在，前端需要与其正确对接。
- 系统分为学生、辅导员、管理员三个角色，各有不同的权限和功能。

## 功能需求
- **FR-1**：学生端删除本地假数据，调用真实查询接口，请求时自动携带学生的院系ID，只查询本院通知和校级通知。
- **FR-2**：学生端移除所有新增、编辑、删除按钮，只保留查看功能。
- **FR-3**：辅导员端列表接口请求时自动携带当前辅导员的院系ID，只查询本院通知。
- **FR-4**：辅导员端发布通知时自动填充本院ID，参数和后端完全对齐。
- **FR-5**：管理员端发布接口添加校级通知标识，请求查询全部校级通知。
- **FR-6**：统一三端接口路径、请求方式、参数字段和后端一致。
- **FR-7**：完善表单校验，确保发布参数完整合法。
- **FR-8**：修复分页参数pageNum、pageSize和后端对齐。
- **FR-9**：按照角色动态控制按钮显示，学生没有操作按钮，辅导员只能操作本院数据，管理员只能操作校级数据。

## 非功能需求
- **NFR-1**：前端性能良好，数据加载速度快。
- **NFR-2**：界面响应及时，用户体验流畅。
- **NFR-3**：错误处理完善，用户得到明确的错误提示。

## 约束
- **技术**：Vue 3 + Element Plus，使用现有API结构。
- **业务**：严格按照角色权限控制，确保数据安全。
- **依赖**：依赖后端API的正确实现。

## 假设
- 后端API已经实现并正常运行。
- 用户信息中包含角色和院系信息。
- 浏览器支持现代JavaScript特性。

## 验收标准

### AC-1：学生端数据渲染
- **Given**：学生登录系统
- **When**：访问通知页面
- **Then**：只显示本院和校级通知，无操作按钮
- **验证**：`programmatic`

### AC-2：辅导员端数据查询
- **Given**：辅导员登录系统
- **When**：访问通知管理页面
- **Then**：只显示本院通知，发布时自动填充本院ID
- **验证**：`programmatic`

### AC-3：管理员端数据管理
- **Given**：管理员登录系统
- **When**：访问通知管理页面
- **Then**：只显示校级通知，发布时自动添加校级标识
- **验证**：`programmatic`

### AC-4：API对接
- **Given**：三端访问系统
- **When**：进行数据操作
- **Then**：API路径、参数、字段名与后端一致
- **验证**：`programmatic`

### AC-5：表单校验
- **Given**：用户发布通知
- **When**：提交表单
- **Then**：表单校验通过，参数完整合法
- **验证**：`programmatic`

### AC-6：分页功能
- **Given**：用户浏览通知列表
- **When**：切换页码或每页条数
- **Then**：分页参数正确，数据正常加载
- **验证**：`programmatic`

### AC-7：权限控制
- **Given**：不同角色用户登录系统
- **When**：访问通知管理页面
- **Then**：根据角色显示相应的操作按钮
- **验证**：`human-judgment`
